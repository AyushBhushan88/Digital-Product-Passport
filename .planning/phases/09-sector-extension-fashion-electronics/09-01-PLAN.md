# Plan: 09-01 - Sector Extension (Fashion & Electronics)

## Goal
Implement specialized schemas for Fashion and Electronics sectors and extend the validation engine to handle category-specific rules.

## Strategy
1. **Define Sector Schemas**: Create new schema files for Fashion (Tier 1-4) and Electronics (BOM).
2. **Refactor Validation Engine**: Transition the `ValidationEngine` from a battery-only logic to a registry-based approach that selects rules based on the product category.
3. **Implement Sector-Specific Rules**:
   - **Fashion**: Mandatory tier data (Origin, Weaving, Dyeing, Assembly).
   - **Electronics**: Mandatory BOM and RoHS/REACH compliance flags.

## Detailed Tasks

### 1. Schema Extensions
- [ ] Create `src/modules/battery/fashion.schema.ts` with Fashion-specific JSON-LD context.
- [ ] Create `src/modules/battery/electronics.schema.ts` with Electronics-specific JSON-LD context.
- [ ] Update `src/modules/battery/battery.types.ts` to include `Category` enum (BATTERY, FASHION, ELECTRONICS).

### 2. Validation Engine Refactoring
- [ ] Refactor `ValidationEngine` to support category-based validation.
- [ ] Implement `validateFashion(data: any): ValidationResult`.
- [ ] Implement `validateElectronics(data: any): ValidationResult`.

### 3. Testing
- [ ] Create `src/modules/compliance/engines/validation.engine.fashion.spec.ts`.
- [ ] Create `src/modules/compliance/engines/validation.engine.electronics.spec.ts`.
- [ ] Add sample data for each sector in `test/fixtures/`.

## Verification Plan
- **Unit Tests**: Ensure `ValidationEngine` correctly identifies missing mandatory fields for Fashion (e.g., `weavingOrigin`) and Electronics (e.g., `bom`).
- **Integration Tests**: Verify that the `IngestionService` can handle multiple product categories correctly using the extended engine.
